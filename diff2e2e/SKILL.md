# 手动测试用例生成器

根据当前分支的代码表更，自动分析修改内容，生成包含主逻辑测试、旁路测试以及边界测试的完整手动测试用例文档。

## 核心原则

核心原则贯穿整个流程，必须严格遵守！

### 原则一：只列与本次改动直接相关的用例

测试用例必须紧扣当前 diff 涉及的功能模块和代码路径。不要为没有变更的页面、组件以及功能生成测试用例。回归测试也仅覆盖与变更存在直接依赖关系的上下游模块。

判断标准：
- **功能测试**：变更直接新增或修改的功能 -> 纳入
- **回归测试**：与变更文件有 import/调用关系的上下游模块 -> 纳入；无直接依赖 -> 不纳入
- **边界测试**：变更涉及的输入/输出/状态边界 -> 纳入；未变更逻辑的边界 -> 不纳入

### 原则二：操作步骤 step by step，每一步都有明确的元素

每条操作步骤必须指明具体的 UI 元素文字（按钮文字、菜单项名称、输入框标签等），使得不了解代码的 QA 或者 AI e2e 测试工具能仅凭步骤描述就能定位元素并完成操作。文案必须从变更组件的**完整源文件**中提取，而非从 Diff 片段猜测。

### 原则三：排除已下线的代码测试

在生成用例前，必须深度递归查询变更代码的引用链路，确认功能确实在线可用后再纳入测试范围。已注释的路由、被 feature flag 关闭的功能、不再挂载的组件等，不得生成对应的测试用例。

### 原则四：测试步骤基于真实用户操作，预期结果仅包含用户可见验证

E2E 测试模拟真实用户行为，测试用例的每一步必须是普通用户在正常使用产品时会执行的动作，预期结果必须是用户无需借助开发者工具就能直接观察到的 UI 变化：

- **禁止代码级检测**：不得出现"打开 DevTools 查看 CSS 属性/DOM 结构"之类的步骤。CSS 属性、元素层级、内联样式等是实现细节，不是用户可见的功能行为。
- **禁止人工构造前置状态**：不得通过修改 DOM、注入脚本、给元素添加兄弟节点等方式制造测试前提。若某用例需要特定数据前提（如"已有 10 个节点池"），在「前置条件」字段中声明即可。
- **允许接口验证**：通过 Network 面板查看 API 请求参数和响应数据是合法的 E2E 验证手段。

## 核心流程

### 步骤 1 + 2：运行 diff 分析脚本

使用内置脚本一次性完成分支检测、主分支识别和 diff 分析：

```bash
bash <SKILL_ROOT>/scripts/git-diff-files.sh
```

脚本自动完成：检测当前/主分支 → 计算 merge base → 收集变更文件 → 过滤 lock/二进制 → 按路径推断模块 → 输出 `.diff-files.json`

也支持手动指定主分支：`bash <SKILL_ROOT>/scripts/git-diff-files.sh master`

脚本完成后，读取 `.diff-files.json` 获取结构化变更信息（`mainBranch`、`currentBranch`、`branchFeature`、`outputPath`、`files`  数组），然后逐个获取关键文件的 diff：

```bash
git diff <主分支>...<当前分支> -- <文件路径>
```

### 步骤 2.5：路由手册检查与生成

在进入变更分析前，检查 `references/page-routes.md` 是否包含有效的路由条目。

**判断标准**：读取文件内容，如果文件为空、只有标题/说明文字而没有任何 Markdown 表格（即没有 `|` 分隔的路由映射行），则视为"无内容"。

**如果文件已有路由条目**：跳过此步骤，直接进入步骤 3。

**如果文件无内容**，按以下流程从项目源码自动生成路由手册：

1. **扫描路由配置文件**：在项目中搜索路由定义文件，常见位置和命名：
   - `src/router/`、`src/routes/`、`config/routes`
   - `router.ts`、`router.js`、`routes.ts`、`routes.js`、`route.config.*`
   - 框架特定模式：Next.js 的 `app/` 或 `pages/` 目录、Vue Router 的 `router/index.ts`、umi 的 `.umirc.ts` 或 `config/routes.ts`

2. **扫描菜单/导航配置文件**：搜索侧边栏或导航菜单的配置源：
   - `menu.ts`、`menu.config.*`、`nav.config.*`、`sidebar.*`
   - 包含 `menuConfig`、`navConfig`、`sidebarConfig` 等变量导出的文件
   - 搜索方式：`grep -rl "menu\|Menu\|sidebar\|Sidebar\|nav\|Nav" src/ --include="*.ts" --include="*.tsx" --include="*.js" --include="*.jsx" | head -20`

3. **解析路由与菜单层级关系**：从路由配置中提取：
   - 路由名称（如 `clusterView.nodePool`、`/dashboard/settings`）
   - 路由路径和嵌套关系（父子路由层级）
   - 对应的页面组件路径

   从菜单配置中提取：
   - 菜单项的显示文案（中文名称）
   - 菜单层级（一级菜单 → 二级菜单 → 三级菜单）
   - 菜单项与路由的绑定关系

4. **生成路由手册**：按以下格式写入 `references/page-routes.md`：

   ```markdown
   # 页面路由指导手册

   根据代码变更涉及的路由名称，查找对应的菜单位置和页面，用于在操作步骤中生成准确的导航指引。

   ## 使用方式

   1. 从 diff 分析中识别出涉及的路由名称
   2. 在本手册中查找该路由对应的菜单路径
   3. 将菜单路径转化为操作步骤中的导航指引文字

   ---

   ## [菜单区域名称]

   | 菜单项 | 路由名 | 导航指引写法 |
   |:---|:---|:---|
   | [菜单显示文案] | `[路由名称]` | [完整的导航操作描述] |
   ```

   **导航指引写法的生成规则**：
   - 一级菜单项：`在左侧导航栏点击「XX」`
   - 二级菜单项（需展开父级）：`在左侧导航栏展开「父菜单」，点击「XX」`
   - 三级或更深：逐级描述展开路径
   - 详情页入口：描述从列表进入详情的方式，如 `在「XX列表」中找到目标项，点击名称进入详情页`

5. **追踪 i18n 文案**：如果菜单项文案使用了国际化 key（如 `t('menu.nodePool')`），追溯到 i18n 文件获取实际中文文案。无法确认时标注 `[文案待确认]`。

6. **生成完成后提示用户**：
   > "已根据项目路由配置自动生成了路由手册（`references/page-routes.md`），包含 XX 个路由条目。请快速浏览确认菜单层级和导航写法是否准确，如有不对的地方请指出。"

**如果未找到路由配置文件**：提示用户手动提供路由信息或跳过路由辅助：
> "未在项目中找到路由配置文件，无法自动生成路由手册。你可以：
> 1. 告诉我路由配置文件的位置，我来解析
> 2. 手动填写 `references/page-routes.md`
> 3. 跳过路由手册，操作步骤中的导航路径将完全从源代码和文档搜索中获取"

### 步骤 3：分析代码变更

> 执行此步骤时，读取 `references/analysis-guide.md` 获取详细的分析操作流程。

对 diff 结果进行结构化分析：

**变更分类**：新增（页面/组件/功能/API）、修改、删除、修复

**模块归类**：将变更文件按功能模块分组，识别核心改动点，分析模块间依赖关系

**改动类型**：纯 UI/样式、业务逻辑、数据流/状态、API 接口、配置/环境、依赖升级

**路由识别**：从变更文件中识别页面路由名称，查阅 `references/page-routes.md` 匹配菜单位置

**已下线代码检测**：路由可达性 → 组件挂载链路 → Feature Flag → 菜单项。确认功能在线后方可纳入测试范围。（详见 `references/analysis-guide.md`）

**UI 元素文案提取**：读取变更组件完整源文件，逐类提取按钮/表单/Tab/弹窗/表格列/提示等文案，追踪 i18n 引用，建立**文案清单**作为后续编写操作步骤的唯一真实来源。（详见 `references/analysis-guide.md`）

**操作路径验证**：从路由配置和菜单配置文件确认导航结构，交叉验证 page-routes.md。（详见 `references/analysis-guide.md`）

**文档辅助验证**：源代码无法确认完整路径时，Web Search 相关文档进行补充。（详见 `references/analysis-guide.md`）

**API 接口变更识别**：当 diff 涉及 API 服务文件（service、request、swagger 等）时，提取接口路径、请求参数、响应结构，建立接口清单。（详见 `references/analysis-guide.md`）

**灰度模块识别**：检查 diff 是否涉及灰度判断代码（灰度函数调用、feature flag、A/B 分支等），提取灰度 key、影响范围、命中与未命中的行为差异，建立灰度清单。（详见 `references/analysis-guide.md`）

### 步骤 4：确认验证环境

向用户询问本次测试使用的验证环境：

> "请提供本次测试的验证环境信息：
> - **环境地址**（如 https://xxx.daily.com）：？
> - **测试账号**（如有特殊账号要求）：？
>
> 如果有多个环境需要验证（如日常 + 预发），请一并提供。"

如果用户跳过，在文档中保留 `[待补充]`。

### 步骤 5：生成测试用例

> 编写操作步骤时，读取 `references/step-writing-guide.md` 获取完整的格式规范和自检流程。

**用例范围筛选**（遵循原则一）：只为以下内容生成用例——diff 直接新增/修改的功能点、与变更有 import/调用依赖的直接上下游、步骤 3 确认为在线可用的功能。

基于变更分析和范围筛选，为每个功能模块生成三类测试用例：

**主逻辑测试（功能测试）**：覆盖核心路径（happy path），验证页面渲染、交互行为、状态流转、数据展示。

**旁路测试（回归测试）**：仅覆盖与变更存在直接依赖关系的已有功能，确认共享组件/公共方法的调用方不受破坏。

**边界测试（边界场景）**：针对变更涉及的输入/输出/状态边界——空数据、超长文本、大量数据、快速重复操作、权限控制、网络异常、并发操作等。

**接口测试（条件生成）**：当步骤 3 识别到 API 接口变更时，为每个变更接口生成接口测试用例，说明触发该接口调用的 UI 操作、请求参数和预期响应值（均来自接口清单）。通过 Network 面板验证实际请求与预期一致。格式见 `references/test-doc-template.md` 接口测试表格。

**灰度场景测试（条件生成）**：当步骤 3 识别到灰度模块时，为每个灰度特性生成一对用例，放入对应功能模块的「功能测试」表格，前置条件写入「测试场景」列开头：
- **【灰度命中】**：前置条件注明"需命中灰度 `<key>`（如使用灰度账号、添加指定参数），验证新功能正常显示和工作
- **【灰度未命中】**：前置条件注明"需未命中灰度 `<key>`，验证降级逻辑正常——旧功能可用、无报错、无样式异常

用例编号规则：灰度用例使用 `{模块前缀}-G{序号}`，如 `A-G1`（命中）、`A-G2`（未命中）。

**操作步骤编写要点**：
- 导航步骤包含完整菜单路径（从哪展开、点击什么、到达哪里）
- 交互步骤指明元素位置和外观（页面右上角的「XX」按钮）
- 验证步骤描述预期看到的具体 UI 变化
- 元素文案必须来自步骤 3 建立的文案清单，不可猜测
- 编写完成后执行自检（文案来源验证、导航路径验证、操作可达性验证、预期结果可观察性验证、逻辑一致性验证）

详细规范和示例见 `references/step-writing-guide.md`。

### 步骤 6：输出测试文档

将测试用例按 `references/test-doc-template.md` 的格式输出。

**输出路径**：`e2e/<分支功能>/MANUAL-TEST-CASES.md`（`branchFeature` 从 `.diff-files.json` 获取）

#### ⛔ 确认点 1：输出前确认（强制，不可跳过）

即使用户在初始指令中已经提供了所有信息（环境地址、分支、测试要求等），也必须在此处暂停并等待用户明确确认后再写入文件。这是为了确保分析结果与用户预期一致，避免生成大量无效用例。

向用户展示分析摘要并等待确认：
> "我已分析了 `<当前分支>` 相对于 `<主分支>` 的 XX 个文件变更，涉及 N 个功能模块：
> 
> - **模块 A**：[核心改动点概述]
> - **模块 B**：[核心改动点概述]
> 
> 即将生成测试文档到 `e2e/<分支功能>/MANUAL-TEST-CASES.md`，包含：
> - 功能测试 XX 条（含灰度场景 XX 条，如有）
> - 回归测试 XX 条
> - 边界测试 XX 条
> 
> 请确认：
> 1. 以上模块和用例范围是否合理？是否有需要增删的模块？
> 2. 确认后我将生成完整测试文档。"

**必须等待用户回复后才能继续。**用户可能要求调整范围、增减模块、修改侧重点。根据反馈调整后再输出。

#### 输出文档

用户确认后，按模板格式写入测试文档文件。

#### ⛔ 确认点 2：输出后审阅（强制，不可跳过）

文档写入后，必须暂停让用户审阅测试用例内容，确认操作步骤和预期结果的准确性后，再进入浏览器自动化阶段。测试用例一旦进入自动化执行，修正成本更高——此处确认是必要的质量关卡。

向用户说明：

> "测试文档已生成到 `e2e/<分支功能>/MANUAL-TEST-CASES.md`。
>
> 请审阅文档中的测试用例，重点关注：
> - 操作步骤是否与实际页面一致
> - 预期结果是否准确
> - 是否有遗漏或多余的用例

> 审阅完成后告诉我，我将进入浏览器自动化测试阶段。如有需要修改的地方，请直接指出。"

**必须等待用户回复后才能进入步骤 7。**用户可能指出文案错误、步骤缺失或用例遗漏，需要先修正文档再继续。

### 步骤 7：浏览器截图与自动化执行（条件执行）

用户确认测试文档无误后，检测当前环境是否有浏览器自动化工具可用（Chrome DevTools MCP、Browser MCP、Playwright 等）。

- **如果不可用**：告知用户当前无法自动截图，建议安装 Chrome DevTools MCP Server（`npx @anthropic-ai/chrome-devtools-mcp-server`）或手动截图。
- **如果可用**：读取 `references/browser-automation.md` 获取完整流程，然后严格按以下节奏对每条用例执行截图：

#### 截图执行节奏（对每条用例重复）

**A. 先写截图计划**：在执行任何浏览器动作前，逐条列出该用例的每个操作步骤是否需要截图，输出为可见文字：

```
截图计划 - 用例 A1（共 4 步）：
- 步骤1 ✓ 截图：打开新页面
- 步骤2 ✓ 截图：弹窗打开
- 步骤3 ✓ 截图：搜索结果变化
- 步骤4 ✗ 不截图：已包含在步骤1-3中
→ 共需截图 3 张
```

**B. 逐步执行，遇到 ✓ 步骤必须截图+写盘后才能继续下一步**：

1. 执行该步骤的浏览器动作
2. 等待 UI 稳定
3. 调用截图工具获取图像
4. 用 Write 工具写入 `e2e/<分支功能>/screenshots/<用例编号>-step<N>.png`
5. 确认写入成功，继续下一步

**C. 截图-预期逐项比对（强制，不可跳过）**：该用例全部步骤执行完毕后、填写自测结果前，必须将「预期结果」拆成独立条目，逐条对照截图中的实际观察写出比对文字。每条必须写明"截图中看到了什么"，矛盾即判失败。只有全部条目通过才能标记 ✅。此流程的详细格式和规则见 `references/browser-automation.md` 第三步。

**D. 回填截图到文档**：该用例执行完毕后，将所有截图填入「页面截图」列。图片 markdown 格式必须以感叹号 `!` 开头，格式为 `![描述](路径)`，例如 `![A1-step1](screenshots/A1-step1.png)`。注意：`[](path)` 是链接不是图片，图片必须写成 `![](path)`，否则图片无法显示。

截图计划是该用例执行的前提——跳过计划直接执行，截图将无法与操作步骤对应，导致关键验证点缺少视觉证据。详细的截图判断标准、失败处理、执行策略见 `references/browser-automation.md`。
