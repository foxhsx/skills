## 测试文档模板

生成的文档必须严格遵循以下模板结构。方括号 `[...]` 内的内容是占位符，需要根据实际分析结果替换为具体内容。`_待补_` 和 `_待执行_` 保持原样，由测试人员后续填写。

### 文档结构概览

文档由以下章节组成，按顺序输出：

1. **背景** — 本次迭代的目标和核心改动描述
2. **影响面** — 变更范围和涉及文件清单
3. **测试信息** — 测试日期、环境、分支、账号等元信息
4. **测试概览** — 用例数量统计表
5. **功能模块测试** — 按模块分章节，每个模块包含功能测试、回归测试、边界场景
6. **通用 UX 回归** — 跨模块的通用体验验证
7. **全局健康检查** — 构建、Lint、Console 等检查项
8. **问题记录** — 测试中发现的问题登记表
9. **Follow-up 技术债** — 本次不修但需跟进的遗留问题

### 模板详情

以下是完整的文档模板（用三个连续反引号括起来的内容即为模板原文）。生成时，将每个 `[占位符]` 替换为从 diff 分析中得出的实际内容，保留所有表格格式和 Markdown 语法不变。

#### 背景章节

```
## 背景

[用 1-3 段话描述本次迭代做了什么、为什么做、解决了什么问题。从 diff 和 commit message 中提炼。]
```

#### 影响面章节

```
## 影响面

### 变更范围

+ **新增**：[从 diff 中识别的新增页面/组件/功能]
+ **修改**：[从 diff 中识别的修改项]
+ **删除**：[从 diff 中识别的删除项]

### 涉及文件清单
```

用例编号规则：
- 功能测试：`{模块前缀}{序号}`，如 A1、A2、B1
- 回归测试：`{模块前缀}-R{序号}`，如 A-R1、B-R1
- 边界测试：`{模块前缀}-E{序号}`，如 A-E1、B-E2
- 模块前缀按章节顺序使用 A、B、C...

操作步骤的导航指引写法，参照 `references/page-routes.md` 中的「导航指引写法」列。
