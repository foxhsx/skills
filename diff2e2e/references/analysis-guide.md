# 步骤 3 详细分析指南

> 在执行步骤 3「分析代码变更」时读取此文件。包含已下线代码检测、UI 元素文案提取、操作路径验证、文档辅助验证的完整操作流程。

---

## 已下线代码检测（遵循核心原则三）

在生成测试用例前，必须排查变更涉及的功能是否仍在线。按以下流程深度递归检查：

1. **路由可达性检查**：从路由配置文件出发，确认变更涉及的页面路由是否被注册且未被注释/条件禁用。如果路由不在当前活跃的路由表中，该页面已下线，不生成用例。

2. **组件挂载链路检查**：从变更的组件出发，沿引用链路向上递归查找（grep/搜索 import 语句），直到找到被路由注册的页面组件为止。如果链路中任何一环断裂（组件不再被任何在线页面引用），该组件已下线。

3. **Feature Flag / 条件渲染检查**：检查变更代码是否被 feature flag、环境变量、A/B 测试开关等包裹。如果功能被禁用（如 `if (false)`、`enabled: false`、被注释），不生成用例。

4. **菜单项检查**：对于涉及导航菜单的变更，确认对应的菜单项在菜单配置中存在且可见（未被 `visible: false` 或条件隐藏）。

具体操作：对每个变更文件，使用 `grep -r` 在项目源码目录中搜索其导出内容的引用情况。如果某个组件/函数在整个项目中没有有效引用（仅在自身文件或测试文件中出现），标记为疑似下线。需要时沿引用链逐层向上递归，直到确认其是否最终接入了活跃的页面路由。

## UI 元素文案提取（遵循核心原则二）

仅从 diff 片段中提取文案是不够的——diff 只展示变更行，不展示按钮、菜单、表单等 UI 元素的完整上下文。必须读取变更组件的**完整源文件**，从完整的渲染逻辑中提取所有 UI 文案。

### 强制执行步骤

1. **读取完整源文件**：对 diff 中涉及 UI 交互的每个组件文件，使用 `cat` 或读取工具获取文件全文（不仅是 diff 片段）。如果组件引用了外部的 constants/config（如菜单配置、表单字段定义），也要追踪读取。

2. **逐类提取并记录来源**：对以下每类元素，记录**文件路径 + 行号 + 原文**三元组：
   - 按钮文字：搜索 `<Button`、`<a-button`、`<XButton` 等组件的 children 文本或 `label`/`title` 属性
   - 表单标签：搜索 `<Form.Item`、`<a-form-item`、`<FormItem` 的 `label` 属性值
   - Tab 名称：搜索 `<Tabs.TabPane`、`<TabPane`、`<a-tab-pane` 的 `tab`/`label` 属性值
   - 弹窗标题：搜索 `<Modal`、`<Drawer`、`<Dialog` 的 `title` 属性值
   - 表格列名：搜索 `columns` 定义数组中的 `title` 字段
   - 占位符文案：搜索 `placeholder` 属性值
   - 提示信息：搜索 `message.success`、`message.error`、`notification`、`Message` 等调用的文案参数
   - 菜单/导航项：搜索菜单配置文件中的 `name`、`title`、`label` 等字段

3. **追踪 i18n 和变量引用**：如果文案是变量引用（如 `{intl.get('key')}`、`t('key')`、`formatMessage({id: 'key'})`），必须追溯到 i18n 文件获取实际中文文案。如果文案来自 props 传入，必须找到调用方确认实际值。

4. **建立文案清单**：将所有提取到的文案整理为结构化清单，格式如下（此清单将在步骤 5 编写操作步骤时作为强制引用源）：

```
## 文案清单
- 按钮「新建」— src/pages/itemList/index.tsx:142
- 按钮「确定」— src/components/CreateModal.tsx:89
- Tab「基本信息」— src/pages/itemList/detail.tsx:56
- 表单项「名称」— src/pages/itemList/CreateForm.tsx:23
- 提示「创建成功」— src/pages/itemList/actions.ts:67
```

将此清单作为后续操作步骤编写的**唯一真实来源**。操作步骤中引用的任何 UI 文案，必须能在此清单中找到对应条目。如果某个操作需要的文案不在清单中，必须回到源文件中重新查找，而不是凭猜测填写。

## 操作路径验证

操作步骤中的页面导航路径是最容易出错的部分。必须从代码层面确认实际的导航结构：

1. **读取路由配置文件**：找到项目的路由配置（如 `router.ts`、`routes.ts`、`config/routes`），确认变更涉及的页面路由确实注册且路径正确。

2. **读取菜单配置文件**：找到侧边栏菜单的配置源（如 `menu.ts`、`nav.config.ts`、`sidebar` 配置），确认菜单层级关系和显示文案。菜单配置中的文案是用户实际看到的导航文字，必须使用这里的文案而非外部参考文档中的近似描述。

3. **交叉验证外部路由参考**：如项目存在外部路由参考文档，可作为参考辅助，但如果源代码中的菜单文案与参考文档不一致，以源代码为准。参考文档可能存在滞后。如果 `page-routes.md` 是本次会话中由步骤 2.5 自动生成的，其内容来源与当前分析的源代码相同，交叉验证意义有限——此时应优先通过产品文档搜索（见下方「文档辅助验证」）或实际页面截图来确认导航路径的准确性。

4. **确认页面层级嵌套**：如果页面有多级嵌套（如 列表页 → 详情页 → 子资源），必须从路由嵌套关系确认每一级的入口方式（是点击列表项进入？还是通过左侧菜单切换？）。

## 文档辅助验证（借助 Web Search 补充操作路径）

源代码提取和路由分析是操作步骤的第一来源，但存在以下场景需要借助产品官方帮助文档进行补充和交叉验证：

- 源代码中的 UI 文案是动态加载的（如从 OpenAPI 返回的菜单配置）
- i18n key 在本地找不到对应翻译文件
- 复杂的多步骤操作流程（如创建主资源、配置子资源），源代码分散在多个文件中难以串联完整路径
- 需要确认操作前置条件（如"需要先开通某服务"、"需要特定权限"）
- 变更涉及的功能有多种入口路径，需要确认主推荐路径

### 执行方式

使用 Web Search 工具搜索产品官方帮助文档：

1. **构建搜索关键词**：根据变更涉及的功能模块，构建针对性的搜索查询。优先使用 `site:<官方文档域名>` 限定搜索范围：
   ```
   搜索示例：
   - "site:<官方文档域名> 新建资源"
   - "site:<官方文档域名> 资源管理"
   - "site:<官方文档域名> 详情页操作"
   - "site:<官方文档域名> 资源升级"
   ```

2. **抓取文档内容**：对搜索结果中官方文档域名的页面，使用 WebFetch 获取文档正文，提取以下信息：
   - **操作步骤**：官方文档中的"操作步骤"章节，通常包含精确的菜单路径和按钮文案
   - **前提条件**：执行操作前需要满足的条件（权限、服务开通、资源准备等）
   - **注意事项**：官方强调的限制条件或易错点，可转化为边界测试用例

3. **交叉验证与优先级**：
   - **源代码文案 > 文档文案**：如果源代码中能明确提取到按钮/标签文案，以源代码为准（文档可能滞后于最新代码）
   - **文档路径 > 猜测路径**：如果源代码中无法确认完整操作路径（如菜单层级），以官方文档描述为准
   - **文档补充源代码空白**：对于源代码中标记了 `[文案待确认]` 的步骤，优先从文档获取准确描述

4. **将文档信息整合到文案清单**：从文档中获取的操作路径信息，标注来源后补充到文案清单中：
   ```
   ## 文案清单（文档补充）
   - 导航路径「资源列表 → 点击资源名称 → 资源管理 → 子资源」— 来源: <官方文档URL>
   - 前置条件「需要拥有 xxx:CreateXxx 权限」— 来源: <官方文档URL>
   - 操作入口「子资源页面右上角，点击"新建"」— 来源: <官方文档URL>
   ```

### 搜索时机

不需要对每个用例都搜索文档。在以下情况触发文档搜索：
- 文案提取阶段遇到无法从源代码确认的文案
- 操作路径自检中发现导航路径不确定
- 涉及复杂的多步骤流程（如资源创建、应用部署等重流程操作）
- 用户要求补充前置条件或操作注意事项

## API 接口变更识别

当 diff 包含 API 相关文件时，提取接口信息，供步骤 5 生成接口测试用例。

### 识别 API 变更文件

以下文件类型的变更通常意味着接口发生了变化：
- API 服务层（`*.service.ts`、`*api.ts`、`*request.ts`、`*/api/*.ts`）
- 接口类型定义（`*.d.ts` 中的 Request/Response interface、`types/api.ts`）
- OpenAPI/Swagger 规范（`swagger.json`、`openapi.yaml`、`*.proto`）
- 请求封装层（axios instance 配置、fetch wrapper）

### 提取接口三要素

对每个 API 变更文件，提取：

1. **接口路径和方法**：从 `axios.post('/api/xxx')`、`request({ url, method })` 等调用中提取
2. **请求参数结构**：从 TypeScript interface/type 定义、函数参数类型、JSDoc 中提取 request body 或 query params
3. **响应数据结构**：从返回类型定义或 `.then(res => res.data)` 的使用方式中推断

### 建立接口清单

将提取结果整理为接口清单，格式与文案清单一致：

```
## 接口清单
- POST /api/resources/{id}/items — src/services/item.ts:45
  请求体：{ name: string, type: string, count: number }
  响应：{ itemId: string, status: string }

- GET /api/resources/{id}/items — src/services/item.ts:30
  查询参数：{ pageSize?: number, pageNum?: number }
  响应：{ items: Item[], total: number }
```

接口清单是步骤 5 生成接口测试用例的唯一真实来源——请求参数和预期返回值必须来自此清单，不可猜测。若 TypeScript 类型定义不完整，优先查找 swagger/proto 文件补充，其次参考已有的请求调用代码中的实际传参。

## 灰度模块识别

当 diff 涉及灰度相关代码时，识别灰度特性的范围和行为，供步骤 5 生成灰度场景测试用例。

### 识别灰度代码模式

以下代码特征表明功能处于灰度状态：

- **灰度函数调用**：`isGray('feature-key')`、`grayManager.isEnabled('xxx')`、`experimentManager.isInGroup('xxx')`、`abtesting.isHit('xxx')`、`featureSwitch.isOn('xxx')` 等
- **灰度配置读取**：从服务端返回的灰度标记字段，如 `config.grayFeatures`、`userConfig.experiments`、`serverConfig.enabledFeatures`
- **条件渲染包裹**：新功能被 `{isGray && <NewFeature />}` 或 `if (isGray) { ... }` 等条件包裹，且旧功能仍保留在 else 分支
- **A/B 分支逻辑**：代码中出现明确的分组判断，不同分组展示不同 UI 或执行不同逻辑

### 提取灰度信息

对每个灰度特性，提取三要素：

1. **灰度 Key**：用于标识该灰度的唯一标识符（如 `'item-list-v2'`、`'enable-new-dashboard'`）
2. **影响范围**：受灰度控制的页面/组件/功能区域
3. **行为差异**：命中灰度时展示/执行什么，未命中时的降级逻辑是什么

### 建立灰度清单

```
## 灰度清单
- 灰度 Key: item-list-v2 — src/pages/itemList/index.tsx:23
  影响范围：资源列表页
  命中：显示新版列表（含「类型」列和批量操作栏）
  未命中：显示旧版列表（无「类型」列，无批量操作栏）
```

灰度清单是步骤 5 生成灰度场景用例的依据——命中/未命中的预期行为必须来自此清单的代码分析，不可猜测。如果降级逻辑不清晰（如 else 分支缺失），需明确标注"未命中行为待确认"，并在确认点 1 向用户说明。
