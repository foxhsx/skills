# 浏览器截图与自动化执行指南

> 在步骤 7 中，当检测到浏览器自动化工具可用时读取此文件。包含工具检测、截图流程、自动化执行、Smoke 检查和结果回填的完整流程。

---

## 7.1 检测可用的浏览器工具

按优先级依次检测当前环境可用的浏览器自动化能力：

1. **Chrome DevTools MCP 工具**（最高优先）：检查是否有 `mcp__chrome-devtools-mcp-server__` 前缀的工具可用（如 `take_screenshot`、`navigate_page`、`take_snapshot` 等）
2. **Browser MCP 工具**：检查是否有 `mcp__browser__`、`mcp__browserbase__`、`mcp__puppeteer__` 等前缀的工具可用
3. **Playwright CLI**：检查 `npx playwright` 是否可执行
4. **Puppeteer 脚本**：检查项目中是否有 puppeteer 依赖

**如果浏览器工具可用**：告知用户并确认测试环境 URL 已就绪，然后进入截图和执行流程。

**如果浏览器工具不可用**，提供备选方案：
> "当前环境没有检测到浏览器自动化工具，无法自动截图。建议：
> 1. **推荐**：安装 Chrome DevTools MCP Server 后重试（`npx @anthropic-ai/chrome-devtools-mcp-server`）
> 2. **手动截图**：按截图指引手动截取每条用例的关键页面，保存到 `e2e/<分支功能>/screenshots/` 目录"

无浏览器工具时，在测试文档的「页面截图」列填入截图指引：`📷 [截图指引: 导航至XX页面，展示XX区域]`

## 7.2 截图流程

即使不进行完整的自动化测试执行，也应该为每条用例截取关键页面截图。截图的核心价值是**验证操作路径的正确性**。

### 截图策略：以预期结果为锚点判断是否截图

截图的目的是为**预期结果提供视觉证据**，而非为每个步骤机械截图。执行每条用例前，先浏览全部操作步骤，判断哪些步骤需要截图：

**需要截图**——步骤执行后页面产生了独立的、需要单独记录的可观察状态：
- 导航到新页面或新视图
- 打开 / 关闭弹窗、抽屉、面板等 UI 元素
- 触发搜索、过滤、排序，使列表或展示内容发生变化
- 步骤的预期结果是一个需要作为独立证据的特定 UI 状态

**不需要单独截图**——
- 步骤的预期结果已隐含在其他步骤的截图中（例如「全过程外层无滚动条」可在其他截图里一并观察，无需额外截图）
- 步骤是被动观察 / 连续性检查，而非产生新状态的交互
- 步骤是中间过渡动作，状态只有完成后续步骤才可见

### 截图保存规则

- 保存路径：`e2e/<分支功能>/screenshots/<用例编号>-step<N>.png`（N 为对应步骤编号）
- 失败截图：`<用例编号>-failed.png`
- 跳过截图的步骤编号不连续是正常的（如只有 step1、step2、step3，没有 step4）

截图完成后，将该用例的全部截图**换行排列**填入测试文档「页面截图」列（详见 7.6）。

## 7.3 解析用例为可执行动作

将测试文档中每条用例的「操作步骤」列解析为结构化的浏览器动作序列：

| 操作步骤描述模式 | 浏览器动作 |
|:---|:---|
| `在左侧导航栏展开「X」，点击「Y」` | click text="X" → click text="Y" |
| `点击页面右上角的「新建」按钮` | click button text="新建" |
| `在「名称」输入框中输入 "test"` | fill label="名称" value="test" |
| `在「类型」下拉框中选择 "标准型"` | click label="类型" → click option="标准型" |
| `点击 Tab 栏中的「详情」标签页` | click tab text="详情" |
| `页面顶部出现绿色提示条"创建成功"` | assert text="创建成功" visible |

**页面跳转必须使用 click 事件**：页面间的导航（包括侧边栏菜单、面包屑、链接跳转等）应通过点击 UI 元素完成，而不是直接调用 `navigate(url)` 跳转。这是因为测试用例模拟的是真实用户行为——用户不会直接输入 URL，他们通过点击菜单和链接来跳转。直接 navigate 会绕过路由拦截、菜单高亮、权限校验等中间逻辑，可能掩盖真实问题。只有在"进入测试起点"（即第一次打开目标页面）时，才允许使用 navigate 直达初始 URL。

## 7.4 逐条执行并收集结果

对每条用例，**先写出截图计划，再执行**。

### 第一步：写出截图计划（必须，不可跳过）

执行任何浏览器动作之前，逐条列出每个操作步骤是否需要截图及原因，格式如下：

```
截图计划 - 用例 A1：
- 步骤1 ✓ 截图：打开新页面，需独立记录页面状态
- 步骤2 ✓ 截图：弹窗打开，需记录打开状态
- 步骤2(关闭) ✓ 截图：弹窗关闭，需记录关闭状态
- 步骤3 ✓ 截图：搜索结果变化，需记录过滤后的列表
- 步骤4 ✗ 不截图：「无滚动条」是全局属性，步骤1-3的截图中已可观察，无需单独截图
共需截图：4张（step1, step2-open, step2-close, step3）
```

这一步必须产生可见文字输出——写出来的计划决定了后续实际截哪些图。如果跳过这一步直接执行，截图将无法与操作步骤对应，导致关键验证点缺少视觉证据，测试结果无法追溯。

### 第二步：按计划逐步执行，截图作为检查点

导航到目标页面，等待加载完成，然后**逐步**执行操作，每一步都按以下节奏处理：

对于计划中标记 **✗ 不截图** 的步骤：
> 执行该步骤的浏览器动作 → 等待 UI 稳定 → 继续下一步

对于计划中标记 **✓ 截图** 的步骤，截图是该步骤的**完成标志**，必须在继续下一步之前完成：
> 1. 执行该步骤的浏览器动作
> 2. 等待 UI 稳定（无 pending 请求、无动画）
> 3. 调用截图工具获取图像
> 4. 用 Write 工具将图像写入磁盘：`e2e/<分支功能>/screenshots/<用例编号>-step<N>.png`
> 5. 确认文件写入成功后，才进入下一步

截图写盘失败时，不得跳过——重试或记录原因后暂停，不能以"截图已完成"为由跳过写盘。

完成所有步骤后，执行**截图-预期逐项比对**（见下方第三步），然后：
- 如该用例在「接口测试」表格中有对应条目：检查网络拦截记录，将捕获到的请求参数和响应回填「实际结果」列
- 检查 Console 是否有 JS Error

### 第三步：截图-预期逐项比对（强制，不可跳过）

这是判定用例通过/失败的唯一依据。跳过此步骤或凭直觉填写结果是严重错误——曾出现过"预期无滚动条，截图中明显有滚动条，却标记为通过"的幻觉问题。为彻底杜绝这类错误，必须执行以下流程：

**对每条用例，在所有截图完成后、填写自测结果前，必须产出以下格式的逐项比对文字：**

```
验证比对 - 用例 A1：

预期1：「页面顶部出现绿色提示"创建成功"」
  → 截图 step3 中实际观察到：页面顶部有绿色 Toast 提示，文案为"创建成功"
  → 判定：✅ 通过

预期2：「全过程外层无滚动条」
  → 截图 step1 中实际观察到：页面右侧存在纵向滚动条
  → 判定：❌ 失败（截图中可见滚动条，与"无滚动条"预期矛盾）

预期3：「列表新增一行，状态为"初始化中"」
  → 截图 step3 中实际观察到：列表最后一行名称为 test-item-01，状态列显示"初始化中"
  → 判定：✅ 通过

综合判定：❌ 失败（预期2 未通过）
失败原因：截图显示页面存在滚动条，与预期"无滚动条"不符
```

**比对规则：**

1. **拆解预期**：将用例的「预期结果」列拆成独立的可验证条目（每个条目是一个具体的 UI 断言）
2. **逐条对证**：对每个预期条目，指出在哪张截图中观察到了什么。必须用具体描述词——"看到了 XX"或"没有看到 XX"，不得使用"正常"、"符合预期"等模糊措辞
3. **矛盾即失败**：如果截图中观察到的状态与预期条目相矛盾（例如预期"无滚动条"但看到了滚动条，预期"显示 5 行"但只有 3 行），该条目必须判定为 ❌，不论其他条目是否通过
4. **无法确认不等于通过**：如果截图中无法确认某个预期（截图范围不够、元素被遮挡、图片不清晰），该条目判定为 ⚠️ 环境受限，不得判定为 ✅
5. **综合判定**：只有全部预期条目都为 ✅ 时，用例才能标记为 ✅ 通过。任何一条为 ❌ 则整体为 ❌，任何一条为 ⚠️（且无 ❌）则整体为 ⚠️

**这段逐项比对文字是该用例通过与否的审计证据——必须输出为可见文字，不得在内部推理中完成后直接写结论。**

### 失败处理

如果某步操作失败（元素未找到、文案不匹配等），这极可能说明操作步骤的描述有误：
- 截图记录失败现场
- 使用浏览器快照（snapshot）查看页面实际的元素文案
- 对比操作步骤中的文案与页面实际文案
- **自动修正操作步骤**：将错误的文案替换为页面上的真实文案，并更新测试文档
- 在执行报告中记录修正内容

### 执行策略

- 每条用例独立执行，不依赖前一条的页面状态（每次从首页或目标页面重新导航）
- 单步操作超时 10s 未响应，标记该步骤为 ⚠️ 并继续下一步
- 单条用例整体超时 60s，标记为 ⚠️ 环境受限
- 遇到登录拦截，暂停并提示用户完成登录后继续
- 文案与页面实际不匹配时，视为步骤描述错误而非测试失败，触发自动修正

## 7.5 执行 Smoke 检查

在逐条用例执行之外，对测试环境进行一次整体健康检查：

```
1. Console 错误收集：过滤所有页面的 console.error（排除已知的第三方 SDK 噪声）
2. 网络请求检查：标记所有 4xx/5xx 响应
3. 性能指标采样：LCP、CLS、INP（如果浏览器工具支持 Performance API）
4. 死链检测：验证页面内所有内链可达（status 200/304）
```

## 7.6 回填执行结果

执行完成后，自动更新测试文档：

1. 将每条用例的「自测 ✅/❌」列从 `_待执行_` 更新为实际结果——此结果**必须来自第三步"截图-预期逐项比对"的综合判定**，不得跳过比对直接填写。如果某用例尚未完成逐项比对，其自测列保持 `_待执行_` 不变
2. 将该用例的所有截图以图片 Markdown 语法**换行排列**填入「页面截图」列，例如：
   ```
   ![A1-step1](screenshots/A1-step1.png)
   ![A1-step2](screenshots/A1-step2.png)
   ![A1-step3](screenshots/A1-step3.png)
   ```
   注意：`!` 不可省略；路径使用相对于测试文档的相对路径（`screenshots/xxx.png`，不含 `e2e/` 前缀）；无截图的步骤不填入
3. 更新「测试概览」表的通过/失败/环境受限数量和通过率
4. 将执行中发现的问题填入「问题记录」表
5. 将 Console Error、网络异常等填入「全局健康检查」的自测结果列

### 执行报告摘要模板

```
## 自动化执行结果

- 执行时间：YYYY-MM-DD HH:mm
- 总用例数：XX
- ✅ 通过：XX
- ❌ 失败：XX
- ⚠️ 环境受限：XX
- 通过率：XX%

### 失败用例
| 用例 | 失败步骤 | 原因 |
|:---|:---|:---|
| A1 | 步骤 3：点击「创建」按钮 | 按钮处于 disabled 状态 |

### Console Errors
- [页面URL] TypeError: Cannot read property 'x' of undefined

### 建议
- [具体修复建议]
```

将此报告追加到测试文档末尾，并在终端输出摘要供用户快速查看。
