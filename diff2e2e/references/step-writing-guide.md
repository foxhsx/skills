# 操作步骤编写规范与自检流程

> 在步骤 5 编写测试用例的操作步骤时读取此文件。包含操作步骤的格式规范、正误示例、以及编写完成后的强制自检流程。

---

## 编写规范

### 1. 导航步骤必须包含完整的菜单路径指引

不要写模糊的页面名称，要写出从哪里点击、展开哪个菜单、最终到达哪个页面：

- ❌ 错误：`打开分类列表页面`
- ✅ 正确：`在左侧导航栏展开「知识库」，点击「分类」进入分类列表页面`

- ❌ 错误：`进入应用列表`
- ✅ 正确：`在左侧导航栏展开「资源管理」，点击「应用」进入应用列表页面`

- ❌ 错误：`查看项目详情`
- ✅ 正确：`在「项目列表」中找到目标项目，点击项目名称进入项目管理页面，在左侧导航栏点击「详情」`

### 2. 交互步骤必须指明操作对象的位置和外观

- ❌ 错误：`点击创建按钮`
- ✅ 正确：`点击页面右上角的「创建分组」按钮`

- ❌ 错误：`填写表单`
- ✅ 正确：`在「分组名称」输入框中输入 "test-group-01"，在「分组类型」下拉框中选择 "标准型"`

- ❌ 错误：`切换 Tab`
- ✅ 正确：`点击页面顶部 Tab 栏中的「YAML」标签页`

方位词（左上角/右上角等）必须能从源代码的 UI 结构（ActionBar 位置、Button 所在 header/footer 区域）或 `references/page-routes.md` 中验证。无法确认时，标注 `[位置待确认]` 而非猜测方位。

### 3. 验证步骤必须描述预期看到的具体内容

- ❌ 错误：`确认创建成功`
- ✅ 正确：`页面顶部出现绿色提示条"分组创建成功"，分组列表中新增一行，名称为 "test-group-01"，状态显示为「初始化中」`

### 4. 多步骤操作用编号分步描述

```
1. 在左侧导航栏展开「资源管理」，点击「应用」进入应用列表页面
2. 点击列表右上角的「创建」按钮
3. 在弹出的表单中填写测试用的应用配置信息
4. 点击表单下方的「确定」按钮
5. 页面自动跳转至该应用的详情页，顶部显示名称与表单中填写的一致
```

### 5. 元素文案必须来自源代码，而非猜测

操作步骤中使用的按钮名、标签名、菜单项名等必须是步骤 3「UI 元素文案提取」阶段从源代码中提取的真实文案。AI e2e 测试工具依赖精确的元素文案来定位页面元素，猜测或编造的文案会导致元素定位失败。

- ❌ 错误（猜测）：`点击「提交」按钮`（实际代码中按钮文案是「确定」）
- ✅ 正确（从代码提取）：`点击「确定」按钮`（对应代码 `<Button type="primary">确定</Button>`）

如果某个元素的文案无法从 diff 中确定（例如是从远端配置加载的动态文案），在步骤中标注 `[文案待确认]` 并说明原因。

### 6. 预期结果必须基于用户可见的 UI 状态，禁止代码级验证

E2E 测试的预期结果只能是用户在正常使用中能直接观察到的现象：

- ❌ 禁止：`打开 DevTools，检查 .full-screen-layout > .next-page 的 style.height 值`
- ❌ 禁止：`在 Console 中执行 JS 确认内部状态`
- ✅ 允许：`页面顶部出现绿色提示「操作成功」`
- ✅ 允许：`通过 Network 面板确认请求参数中包含 groupType: "standard"`

理由：E2E 测试模拟真实用户视角——CSS 属性等实现细节随布局变化极易失效，且用户无法感知。通过 Network 面板检查 API 请求/响应参数是合法的 E2E 验证手段。

### 7. 操作步骤只包含真实用户的自然交互，禁止人工构造前置状态

不得在步骤中通过人工手段制造测试环境：

- ❌ 禁止：`通过给 .page-container 添加兄弟元素触发布局变化`
- ❌ 禁止：`在 Console 中修改 window.__feature_flags__ 开启某功能`
- ✅ 正确：在「前置条件」字段中声明所需的数据状态（如"项目中已有 ≥ 10 个分组"），通过正常 UI 操作使页面进入所需状态

### 8. 操作步骤自检（必须执行）

每条用例的操作步骤编写完成后，必须执行以下自检流程。跳过自检是产出质量差的主要原因。

---

## 自检流程

对每条操作步骤中出现的 UI 元素文案（用「」包裹的文字），执行以下验证：

### a) 文案来源验证

确认该文案出现在步骤 3 建立的「文案清单」中。如果不在清单中，说明这个文案是猜测的——立即回到源文件查找真实文案。

### b) 导航路径验证

对于涉及页面跳转的步骤，确认：
- 起始页面是什么？（用户执行到此步时在哪个页面上？）
- 目标元素在页面的什么位置？（左侧导航栏？页面顶部？表格操作列？）
- 点击后到达的页面是否正确？（路由跳转目标与描述一致？）
- 步骤中的方位词（左上角/右上角等）是否已从源代码或 `references/page-routes.md` 确认？（不得凭感觉猜测方位）

### c) 操作可达性验证

确认用户从上一步的结束状态出发，能自然地执行当前步骤。如果中间有遗漏的步骤（比如需要先展开某个折叠面板才能看到按钮），必须补充。

### d) 预期结果可观察性验证

确认预期结果是用户在页面上能直接看到的变化（提示语、列表新增行、状态变更等），而不是需要打开开发者工具查看 CSS/DOM/JS 内部状态才能验证的内容。通过 Network 面板检查 API 请求参数和响应数据是例外——这属于合法的 E2E 验证手段。

### e) 逻辑一致性验证

确认用例中的前置条件、操作步骤与预期结果在语义上指向同一个测试对象：

- 前置条件的量纲必须与被测对象一致——测试「分组列表」时，前置条件应为"项目中分组数量 ≥ N"，而非"成员数量 ≥ N"（成员数量是某个分组内的成员个数，与分组数量是不同的概念）
- 预期结果必须是当前操作能直接引起的变化，不得混入不相关的状态描述
- 用例标题声明测试"X"时，操作步骤中所有操作对象也应是 X，而非 X 的子项或关联实体

### 自检标注格式（内部使用，不输出到最终文档）

```
步骤: 点击页面右上角的「创建分组」按钮
来源: src/pages/group/index.tsx:142 → <Button type="primary">创建分组</Button>
验证: ✓ 文案匹配 ✓ 位置描述正确（页面右上角 ActionBar 区域）
```

如果自检中发现任何一步无法通过验证，不要输出该用例——回到源文件重新确认后修正。
